# matchlen-wasm — proof-of-concept wasm-SIMD (v128) for go-simd/matchlen

**Status: prototype.** A demonstration that pure-Go SIMD on wasm is achievable
today, even though the Go compiler does not emit `v128` instructions from Go
source. The kernel lives in a small companion WebAssembly module
(`matchlen.wasm`, compiled from `matchlen.wat`) and is called from Go via
`//go:wasmimport`.

## Why this exists

The [go-simd](https://github.com/go-simd) family accelerates 6 arches
(amd64/arm64/riscv64/loong64/ppc64le/s390x) through per-arch Plan 9 assembly
kernels generated by [go-asmgen](https://github.com/go-asmgen/asmgen). The 7th
target Go supports — `GOOS=js GOARCH=wasm` / `GOOS=wasip1 GOARCH=wasm` — has
no Plan 9 asm dialect and no v128 emission from Go source. Result: our wasm
builds fall back to the scalar path.

Wasm-SIMD (`v128`) has been a stable proposal since 2021 and every major
browser + wazero + wasmtime supports it. The **only** thing missing is a way
to write `v128` code from Go. This proto shows the path.

## Architecture

```text
┌─────────────────────────────────────┐
│ Go module  (matchlen_wasm.go)       │
│                                     │
│   //go:wasmimport env matchlen16    │
│   func matchlen16(a,b,n uint32) …   │──── boundary
└─────────────────────────────────────┘        │
                                               ▼
┌─────────────────────────────────────┐
│ Companion wasm module  (matchlen.wat)│
│                                     │
│   (func $matchlen16                 │
│     (v128.load) (i8x16.eq)          │
│     (i8x16.all_true) …)             │
└─────────────────────────────────────┘
```

The two share **the same linear memory** — the WAT declares
`(import "env" "memory" (memory 0))`, so `v128.load` reads the exact bytes
the Go slice sits on. No copy.

## Build

You need [WABT](https://github.com/WebAssembly/wabt)'s `wat2wasm` to compile
`matchlen.wat` → `matchlen.wasm`. Then embed the wasm bytes into the host
runtime (JS or wazero).

```sh
# darwin/arm64:  brew install wabt
# linux:        apt install wabt / dnf install wabt
wat2wasm --enable-simd matchlen.wat -o matchlen.wasm
```

## Host-side wire-up

### JavaScript (browser / node)

```js
const wasmBytes = await fetch("matchlen.wasm").then(r => r.arrayBuffer());
const goInstance = new Go();
const { instance } = await WebAssembly.instantiate(await getGoWasm(), {
  ...goInstance.importObject,
  env: {
    ...goInstance.importObject.env,
    matchlen16: (ptrA, ptrB, n) => {
      // Instantiate the SIMD module lazily, sharing Go's linear memory:
      const { instance: k } = WebAssembly.instantiateSync(wasmBytes, {
        env: { memory: instance.exports.mem }, // Go's exported linear memory
      });
      return k.exports.matchlen16(ptrA, ptrB, n);
    },
  },
});
goInstance.run(instance);
```

### wazero (Go server-side WASI host)

```go
runtime := wazero.NewRuntime(ctx)
kernel, _ := runtime.Instantiate(ctx, wasmBytes)
_, _ = runtime.NewHostModuleBuilder("env").
    NewFunctionBuilder().WithFunc(func(ctx context.Context, m api.Module,
        ptrA, ptrB, n uint32) uint32 {
        return uint32(kernel.ExportedFunction("matchlen16").
            Call(ctx, api.EncodeI32(int32(ptrA)),
                     api.EncodeI32(int32(ptrB)),
                     api.EncodeI32(int32(n)))[0])
    }).Export("matchlen16").
    Instantiate(ctx)
```

## Expected performance

| input size | wasm-SIMD (v128) | scalar Go loop | speedup |
| ---: | ---: | ---: | ---: |
| 16 B | ~40 ns | ~10 ns | ~0.25× (crossover) |
| 64 B | ~50 ns | ~35 ns | ~0.7× |
| 1 KiB | ~200 ns | ~450 ns | ~2.3× |
| 64 KiB | ~10 µs | ~30 µs | ~3× |
| 1 MiB | ~150 µs | ~500 µs | ~3.3× |

Crossover point ~40 bytes. Below that, the fixed wasmimport call overhead
(~30-50 ns on V8/wazero) dominates and the scalar Go loop wins. Above the
crossover, `v128.load` + `i8x16.eq` + `i8x16.all_true` gives roughly a
constant 3× win up to memory bandwidth.

For LZ4 match extension — the primary consumer of `matchlen` — the typical
match tail is 100 B – 2 KiB, so the SIMD path pays.

## Trade-offs vs the native-arch path

| aspect | native (amd64/…/s390x) | wasm-SIMD (v128) via this proto |
| --- | --- | --- |
| ceiling | full arch SIMD (AVX2/VXE2/…) | v128 = 16 B / cycle common denominator |
| overhead per call | 1-2 ns (direct branch to Plan 9 asm) | 30-50 ns (wasmimport boundary crossing) |
| deployment | `go build` and ship | ship `matchlen.wasm` alongside; host wire-up |
| portability | 6 arches × Plan 9 asm | 1 wasm module, runs everywhere wasm-SIMD is stable |

## Roadmap

1. **This proto ships** — `matchlen16` alone. Consumer opts in per-call site.
2. **Automate**: teach `go-asmgen` to emit `.wat` for the kernels it already
   emits for the 6 native arches. That gives us `hex`, `base64`, `bitpack`,
   `utf8`, `popcount`, etc. on wasm-SIMD in one motion.
3. **Native fold**: if the Go compiler ever gains `v128` emission (SIMD
   autovectorization or explicit intrinsics), collapse this scaffolding into
   the compiler-emitted path — same API, no consumer change.

## Non-goals

- No support on `wasm` targets where wasm-SIMD is missing (very old runtimes,
  pre-Chrome 91 / pre-Firefox 89). Feature detection is the host's job.
- No support for `//go:wasmexport` (exporting Go functions to JS) — the flow
  is JS/host → Go → SIMD kernel → back to Go, not the reverse.
- Not a browser sandbox: the SIMD kernel READS memory the Go host wrote; it
  can never escape the shared linear memory the module is instantiated with.
